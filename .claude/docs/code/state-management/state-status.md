## Status communication

System content should answer three questions when needed:

1. what happened
2. what it means
3. what the user can do next

---

## Error handling

Error handling should support recovery, not punish the user.

- place the message near the affected context when possible
- preserve user input when recovery is possible

---

## Success state

Success should confirm completion without forcing users to interpret subtle visual changes.

---

## Empty state

Empty states should explain why the view is empty and suggest the next useful action.

- explain why the space is empty (`No projects yet.`)
- include a useful action if possible

---

## Restricted area

Communicate to user that has no rights to go there.
